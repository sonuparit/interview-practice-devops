This is one of the most common Linux interview questions for DevOps engineers.

The interviewer is **not checking whether you memorized the directories**. They want to know whether you understand **how Linux is organized** and **where to troubleshoot problems**.

A simple way to remember it is:

> **The root filesystem (/) is like a city.**
> Every directory has a specific job. If everything were stored in one place, Linux would become impossible to manage.

---

# 1. /

## What is it?

The root directory.

It is the top-most directory in Linux. Every file and every directory starts from here.

```
/
├── bin
├── boot
├── etc
├── home
├── var
└── usr
```

Think of it as:

> The Earth.

Everything exists somewhere under `/`.

### Why is it needed?

Linux needs one starting point for the entire filesystem.

### DevOps

Whenever you use

```bash
cd /
```

you're standing at the top of the filesystem.

---

# 2. /bin

## What is it?

Contains essential commands every user needs.

Examples

```
ls
cp
mv
cat
echo
rm
```

Check:

```bash
ls /bin
```

### Why is it needed?

Imagine Linux cannot find `ls`.

Then you couldn't even list files.

These are the minimum commands Linux needs to function.

### Purpose

Essential user commands.

### DevOps

Almost every shell script uses commands from here.

Example

```bash
cp
mv
rm
mkdir
```

---

# 3. /sbin

## What is it?

Contains essential **system administration** commands.

Examples

```
fdisk
mkfs
reboot
shutdown
fsck
```

### Why?

Regular users don't usually manage disks or reboot servers.

These commands are mainly for administrators.

### DevOps

Whenever you're fixing production servers you'll often use

```
fsck
mkfs
reboot
shutdown
```

---

# 4. /boot

## What is it?

Contains files required to boot Linux.

Example

```
Kernel
initramfs
GRUB files
```

### Why?

Without these files Linux cannot start.

Think of it as

> BIOS → GRUB → Kernel

Everything needed during startup lives here.

### DevOps

Useful when troubleshooting

* Kernel updates
* GRUB issues
* Boot failures

---

# 5. /dev

## What is it?

Contains device files.

Linux treats hardware as files.

Examples

```
/dev/sda
/dev/sdb
/dev/null
/dev/tty
```

### Why?

Instead of creating different APIs for disks, keyboards, USBs...

Linux exposes everything as files.

### DevOps

Very important.

Example

Disk

```
/dev/sda
```

Create filesystem

```bash
mkfs.ext4 /dev/sdb1
```

Mount

```bash
mount /dev/sdb1 /mnt
```

---

# 6. /etc

## What is it?

Contains system configuration files.

Examples

```
passwd
hosts
fstab
ssh
systemd
```

### Why?

Linux keeps configuration separate from programs.

### DevOps

One of the most important directories.

Examples

SSH config

```
/etc/ssh/
```

DNS

```
/etc/resolv.conf
```

Mounts

```
/etc/fstab
```

Hostname

```
/etc/hostname
```

---

# 7. /home

## What is it?

Users' personal folders.

Example

```
/home/sonu
/home/john
```

### Why?

Every user gets their own workspace.

### DevOps

When users log in

```
cd ~
```

takes them here.

---

# 8. /root

## What is it?

Home directory of the root user.

```
/root
```

### Why?

Root is a special user.

Instead of

```
/home/root
```

Linux gives root its own location.

### DevOps

Many admin scripts run here.

---

# 9. /usr

## What is it?

Contains most installed software.

Examples

```
Python
Git
Docker
gcc
vim
```

Think of it as

> Program Files (Windows)

### Why?

Keeps applications organized.

### DevOps

When installing packages

```
apt install git
```

many files go under

```
/usr/bin
/usr/lib
/usr/share
```

---

# 10. /var

## What is it?

Contains data that changes frequently.

Examples

```
logs
cache
mail
database
spool
```

### Why?

Logs keep growing.

Databases keep changing.

These shouldn't be mixed with program files.

### DevOps

Extremely important.

Logs

```
/var/log
```

Web server

```
/var/www
```

Docker

```
/var/lib/docker
```

Package cache

```
/var/cache
```

---

# 11. /tmp

## What is it?

Temporary files.

### Why?

Programs sometimes need short-term storage.

Example

Browser download

```
temp123.tmp
```

Installer files

```
setup.tmp
```

Usually removed after reboot.

### DevOps

Applications often use `/tmp` during installations or processing.

---

# 12. /proc

## What is it?

Virtual filesystem.

Doesn't exist on disk.

Generated by the kernel.

### Why?

Provides information about the running system.

Examples

CPU

```
/proc/cpuinfo
```

Memory

```
/proc/meminfo
```

Running processes

```
/proc/1234/
```

### DevOps

Useful for monitoring.

Example

```bash
cat /proc/meminfo
```

---

# 13. /sys

## What is it?

Virtual filesystem for kernel and hardware information.

### Why?

Lets Linux expose device information and kernel settings.

### DevOps

Useful for:

* Debugging hardware
* Kernel tuning
* Device information

---

# 14. /run

## What is it?

Stores runtime information.

Examples

PID files

Sockets

Lock files

### Why?

These files only exist while the system is running.

After reboot they disappear.

### DevOps

Systemd uses this heavily.

Example

```
/run/systemd/
```

---

# 15. /mnt

## What is it?

Temporary mount point.

### Why?

Admins mount disks here manually.

Example

```bash
mount /dev/sdb1 /mnt
```

### DevOps

Very common while troubleshooting.

---

# 16. /media

## What is it?

Automatically mounted removable devices.

Example

USB drive

DVD

External HDD

### Why?

Desktop environments mount removable media here automatically.

### DevOps

Useful when copying backups or logs to a USB drive.

---

# 17. /opt

## What is it?

Optional third-party software.

Example

```
/opt/google
/opt/aws-cli
```

### Why?

Keeps manually installed software separate from the operating system.

### DevOps

Many enterprise applications are installed here.

---

# 18. /srv

## What is it?

Contains data served by services.

Examples

Websites

FTP files

Git repositories

### Why?

Separates service data from application binaries.

### DevOps

Sometimes web content is stored here instead of `/var/www`.

---

# 19. /lib and /lib64

## What is it?

Shared libraries used by programs.

Think of them as Linux's equivalent of DLL files on Windows.

### Why?

Instead of every program carrying its own copy of common code, they all share libraries stored here.

### DevOps

If a required library is missing, applications may fail to start with errors like:

```bash
error while loading shared libraries
```

---

# 20. /lost+found

## What is it?

Recovery directory created by filesystems such as ext4.

### Why?

If the system crashes or loses power, filesystem repair tools (like `fsck`) can recover orphaned files and place them here.

### DevOps

Useful after disk corruption or an unexpected shutdown to recover data.

---

# 21. /cdrom

## What is it?

A mount point for CD/DVD media.

### Why?

Historically used to access optical drives.

### DevOps

Rarely used today, but you might still encounter it on older systems or virtual machines.

---

# 22. `swap.img`

## What is it?

A swap file (not a directory).

### Why?

Provides additional virtual memory when RAM is exhausted.

### DevOps

Helps prevent out-of-memory conditions on small systems, though excessive swapping can hurt performance. Monitoring swap usage is an important part of system administration.

---

# ⭐ The interview-friendly summary

| Directory        | Simple meaning                | Common DevOps use                                 |
| ---------------- | ----------------------------- | ------------------------------------------------- |
| `/`              | Root of everything            | Starting point of the filesystem                  |
| `/bin`           | Essential user commands       | Shell scripts, basic utilities                    |
| `/sbin`          | System admin commands         | Disk, network, boot administration                |
| `/boot`          | Boot files                    | Kernel and GRUB troubleshooting                   |
| `/dev`           | Hardware represented as files | Mounting disks, storage management                |
| `/etc`           | Configuration files           | Editing system, SSH, network, and service configs |
| `/home`          | User data                     | User environments                                 |
| `/root`          | Root user's home              | Administrative work                               |
| `/usr`           | Installed software            | Application binaries and libraries                |
| `/var`           | Variable data                 | Logs, Docker data, databases                      |
| `/tmp`           | Temporary files               | Installers and temporary application data         |
| `/proc`          | Kernel/process information    | Performance monitoring and debugging              |
| `/sys`           | Kernel and device interface   | Hardware inspection and kernel tuning             |
| `/run`           | Runtime state                 | PID files, sockets, systemd                       |
| `/mnt`           | Manual mount point            | Temporary disk mounts                             |
| `/media`         | Auto-mounted removable media  | USB and external drives                           |
| `/opt`           | Third-party software          | Vendor-installed applications                     |
| `/srv`           | Service data                  | Web, FTP, or Git server content                   |
| `/lib`, `/lib64` | Shared libraries              | Dependencies for executables                      |
| `/lost+found`    | Recovered filesystem data     | File recovery after crashes                       |
| `/cdrom`         | Optical media mount point     | Legacy systems                                    |
| `swap.img`       | Virtual memory file           | Memory management and OOM prevention              |

### How to answer in an interview

If an interviewer asks, "Can you explain the Linux root directories?", a concise answer could be:

> "Linux organizes files by purpose rather than by drive letters. Configuration files are kept in `/etc`, user data in `/home`, system logs and other changing data in `/var`, installed software primarily in `/usr`, device files in `/dev`, boot-related files in `/boot`, temporary files in `/tmp`, and kernel information is exposed through virtual filesystems like `/proc` and `/sys`. Knowing this layout helps me troubleshoot issues quickly—for example, checking `/var/log` for logs, `/etc` for configuration problems, `/proc` for runtime system information, and `/dev` when working with disks or storage."
